APIs

Computers look like just one machine, especially as they get smaller and smaller. However, they are composed of a few discrete parts. We’ve already seen the hard drive for non-volatile storage, the RAM for volatile storage, and the CPU for processing instructions. But there can also be other components, like graphics cards, sound cards, or networking cards.

Each of these components provide different functionality to the computer and require their own specific code to handle the communication with the CPU. Storing data in RAM is a fairly different process from storing it on the hard drive.

Luckily, we don’t need to know very much about the inner workings of these components. The “operating system”, like Windows, MacOS, or a variety of Linux, provides an abstraction layer that let’s you spend more time thinking about “what” you want to do instead of the particulars of “how” you want to do it.

Taking the hard drive as an example, we can see one of these abstraction layers in action.

From the hard drive’s perspective, it provides “block” storage. That is, it allows the CPU to read or write about 500 kilobytes of data at a time. While a restriction like that makes the hardware easier to contruct and the communication more straightforward, it doesn’t lead to a great user experience. Songs and books and videos wouldn’t fit in just one block, so you the programmer would have to manage breaking them up into smaller blocks to send them to the hard drive for storage and then reassembling them into a single piece of media for use.

The operating system steps in here to solve many of these problems for you. It provides the file system to handle all the underlying block work. That way you can work with an abstract “file” and not worry about the specifics of how the hard drive works.

The file system then provides an “Application Programming Interface”, or API, for you to write programs with. The API is mostly just a bundle of commands that you can send to the file system to create, read, write, or delete files.

That means for each component there are two things to learn. First, you have to understand the abstraction layer. What problem is it solving? How is it intended to be used? Second, you have to learn the particulars of the API. How do you create and manipulate new data? What are the different options available to you?

Keeping all of these layers separate in your mind can be difficult and even more trouble than it is worth at times. However, if you can successfully do it, you can often pinpoint issues that come up with your programs. You also get a better view on the possibilities for new ideas. You can quickly get to the details of creating a new type of hard drive or a new type of file.

The various APIs for memory, storage, and networking form the foundations of programming. So we are going to cover the theory behind the APIs as well as the practical usage of them. Later on, we can go back and dig deeper into how the lower layers like the file system and operating system actually work.
